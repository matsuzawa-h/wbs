1. システム概要 / 背景

テンプレートファイル.xls（工程マイルストーン Ver1.13、Excel + VBA アドイン km_module.xla）を Web 化したもの。 社内 LAN 上で複数 PC から同じ DB を参照しつつガント形式でプロジェクト進行を管理する。

項目内容
対象環境Windows 11 上の Node.js プロセス（Windows サービスとして常駐）
公開範囲社内 LAN（インターネット非公開、認証は MVP 範囲外）
ストレージSQLite ファイル DB（C:\AppsData\WbsWeb\wbs.db）
同時編集MVP: 単独編集前提（将来 Socket.IO で共同編集に拡張）
主言語TypeScript（フロント・バック共通）

progressINTEGER0–100、default 0
assignee_idINTEGERFK → assignees.id (ON DELETE SET NULL)
statusTEXTdefault ''（自由記述、状態名）
sort_orderINTEGERNOT NULL, default 0（行 DnD で更新）

インデックス: (project_id), (parent_id), (project_id, sort_order).

予定 vs 実績の集約ルール

項目level=3 (項目)level=1, 2 (集約)
start_date (予定開始)直接編集min(子.start_date)
end_date (予定終了)start_date + duration の平日カウントmax(子.end_date)
duration (予定日数)直接編集（平日換算）businessDaysBetween(min start, max end)
actual_start_date (実績開始)直接編集（NULL 可）min(子.actual_start_date) — null は除外
actual_end_date (実績終了)直接編集（NULL 可）max(子.actual_end_date) — null は除外
planned_hours (予定工数)直接編集（NULL 可、時間単位）sum(子.planned_hours) — null は除外
actual_hours (実績工数)直接編集（NULL 可、時間単位）sum(子.actual_hours) — null は除外
日付カスケード（後続タスクのシフト）は 予定 end_date が変わったときだけ発火します。実績日や工数を変更しても、後続の予定日はずれません（実績は過去の事実なので）。

6. 日付カスケード仕様

営業日の定義

  • 営業日 = 月〜金（土日をスキップ）。祝日マスタは MVP 範囲外。
  • 日付は YYYY-MM-DD 文字列で扱い、UTC 基準でパースしてタイムゾーン差分を排除。

終了日の算出

endDate = addBusinessDays(startDate, duration - 1)。終了日は inclusive（duration=1 は同日終了）。

カスケード動作（チェーン伝播）

カスケードは「重なる」または「連日（直後に始まる作業）」のときだけ伝播し、ギャップがあるタスクで停止します。一律に全部シフトしません。

  1. level=3 タスク T の startDate または duration が変更され、endDate が変動。
  2. 同じ親（中項目）配下かつ sortOrder > T.sortOrder の level=3 タスクを sortOrder 昇順で取得（他の中項目配下は対象外）。
  3. 各タスク B について、次のいずれかを満たすときだけシフト：
    • 連日（contiguous）: B の startDate ≤ 直前タスクの元 endDate の翌営業日（チェーン内で直接続いていた）
    • 重なる（overlap）: B の startDate ≤ チェーンの新しい endDate（押された結果ぶつかる）
    どちらにも当てはまらなければ ギャップありと判断してチェーンを停止。以降は触らない。
  4. シフト先は nextBusinessDay(チェーンの新しい endDate)。期間（duration）は維持して endDate を再計算。
  5. 影響を受けた集約行（level=1 と level=2）を「子の最小開始日 / 最大終了日」で再集計。
  6. すべて単一 db.transaction() 内で完了。
例: A=月〜水、B=木（連日）、C=翌々週月（ギャップあり）の場合。A を金まで伸ばすと B は土日越えで来週月〜火にシフト、C は触らない。

連動シフトの無効化（UI 側のトグル）

ヘッダーの「🔗 連動 ON / 🔓 連動 OFF」トグルで、ユーザーは編集時のカスケードを一時的に無効化できる。 OFF にした状態で予定日を変更すると、その 1 タスクだけが書き換わり、同じ中項目の後続タスクは影響を受けない。 内部的には PATCH /api/tasks/:id に cascade: false を付与している。設定は localStorage に保存される。

例: 金曜開始 + 期間 3 平日 → 終了日は翌週火曜。
バーを 3 平日後ろにドラッグすると、後続の level=3 タスクが +3 平日 ずれ、土日は数えない。

7. デプロイ手順

事前準備（PC ごとに 1 度）

  • Node.js 20 LTS をインストール
  • corepack enable で pnpm を有効化
  • NSSM を C:\Tools\nssm.exe に配置
  • 管理者 PowerShell を起動できる権限

初回デプロイ

cd c:\Git\WBS\WbsWeb
git pull
.\scripts\deploy.ps1                      # AppDir=C:\Apps\WbsWeb, DataDir=C:\AppsData\WbsWeb
.\scripts\install-service.ps1             # 管理者 PowerShell が必須

2 回目以降の更新リリース

git pull
.\scripts\restart-service.ps1             # stop → deploy → migrate → start

配置後ディレクトリ

C:\Apps\WbsWeb\                  ← アプリ本体（更新で書換）
├── dist\main.js
├── node_modules\
└── public\                       ← Vue ビルド成果物

C:\AppsData\WbsWeb\               ← 永続データ（更新で絶対に触らない）
├── wbs.db
├── wbs.db-wal / wbs.db-shm        ← WAL モード
├── logs\
│   ├── stdout.log
│   └── stderr.log
└── backups\
    └── wbs_YYYYMMDD_HHmmss.db

バックアップ

サービスを停止せず安全にスナップショットを取得できる：

.\scripts\backup.ps1
# → C:\AppsData\WbsWeb\backups\wbs_YYYYMMDD_HHmmss.db

復元はバックアップを wbs.db にリネーム後サービスを再起動。

8. 受入テストシナリオ

  1. 開発機で pnpm dev:api + pnpm dev:web を起動 → http://localhost:5173 を開く
  2. プロジェクトを 2 件作成 → 切替できる
  3. プロジェクト A に 大項目1 ⊃ 中項目1 ⊃ 項目を 3 件 を登録 → 3 階層で表示される
  4. 金曜開始 + 期間 3 平日 の項目を作成 → 終了日が翌週火曜（土日スキップ）
  5. 中間タスクのバーを 3 平日後ろにドラッグ → 後続項目が +3 平日シフト、大項目/中項目の集約日付も再計算
  6. 行をドラッグして並べ替え → DB に sortOrder が保存される
  7. ブラウザを再読み込み → 状態が永続化されている
  8. deploy.ps1 → install-service.ps1 実行 → 別 PC のブラウザから http://<このPCのIP>:5000 でアクセスできる
  9. PC 再起動 → サービスが自動起動して継続利用できる
  10. backup.ps1 実行中にアプリを操作 → バックアップが正常に取れて、コピー復元後も同じ状態が再現できる

9. 既知の制限 / 将来拡張ロードマップ

既知の制限（MVP 段階）

  • 同時編集すると最後の更新が勝つ（楽観ロックなし）
  • 祝日は考慮されない（土日のみスキップ）
  • 認証なし。社内 LAN 前提で IP 制限のみ
  • 日本語固定（多言語化未実装）
  • 印刷整形は CSS の @media print のみで、PDF 出力機能は別途
