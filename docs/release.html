1. 前提・構成図

項目内容
本番マシンWindows 11 PC 1 台（社内 LAN）
同時利用10 ユーザ想定（SQLite で問題なし）
URLhttp://<本番マシンの IP>:5000/ （HTTP / 平文）
常駐方式Windows タスクスケジューラ（追加インストール不要）、OS 再起動で自動起動
MCP モジュール同梱（同一マシンからのみアクセス可、Bearer トークン必須）
既存データ開発機の SQLite を丸ごと持ち込み

配置イメージ

本番マシン（Windows 11）
├── C:\Apps\WbsWeb\               ← アプリ本体（更新で書き換わる）
│   ├── dist\main.js
│   ├── node_modules\
│   └── public\                   ← Vue ビルド成果物
└── C:\AppsData\WbsWeb\           ← 永続データ（更新でも触らない）
    ├── wbs.db                    ← SQLite データ
    ├── wbs.db-wal / -shm         ← SQLite ジャーナル
    ├── start.cmd                 ← タスクスケジューラから呼ばれる起動スクリプト
    │                                (環境変数 + ログリダイレクト + node 起動)
    ├── templates\
    │   └── テンプレートファイル.xls   ← Excel 出力／取込で使う旧テンプレ

📌 役割の表示 以降の手順では、操作するマシンを DEV（開発機）／PROD（本番マシン）／共通 で示します。

PROD2. 本番マシン側 ワンタイム準備

初回のみ、本番マシン上で以下を実施します。すべて管理者として PowerShell を開いて実行してください。

2.1Node.js 20 LTS のインストール

  1. ブラウザで https://nodejs.org/ を開く
  2. 「LTS」（推奨版）の Windows Installer（.msi）をダウンロード
  3. インストーラーを起動 → 既定設定で完了（PATH 追加にチェックが入っていることを確認）
  4. PowerShell を一度閉じて開き直し、確認: 
    PS> node --version
v20.x.x
PS> npm --version
10.x.x

2.2pnpm を有効化

PS> corepack enable
PS> pnpm --version
9.x.x
💡 corepack Node.js 16.10 以降に同梱されている公式のパッケージマネージャ管理ツールです。pnpm を別途インストールせずに使えます。
⚠ npm.ps1 を読み込むことができません エラーが出る場合 Windows のクリーン環境では PowerShell の ExecutionPolicy が Restricted になっていて、Node.js 同梱の npm.ps1 / pnpm.ps1 が実行できません。下記を 1 回実行 → PowerShell を再起動してください: 
PS> Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
確認プロンプトには Y で答えます。-Scope CurrentUser なので、サービスを実行する SYSTEM アカウントには影響しません（タスクスケジューラから直接 node.exe を呼ぶため ExecutionPolicy 非依存）。

2.3常駐方式：Windows タスクスケジューラを使う

本構成では 追加ソフトを入れず、Windows 標準のタスクスケジューラで Node.js プロセスを常駐させます。NSSM などの 3rd party サービスマネージャは不要です。

項目方式
起動トリガシステム起動時（At startup）
実行アカウントSYSTEM（ユーザがログオンしていなくても動く）
環境変数 / ログリダイレクトラッパー start.cmd 経由（5.2 で作成）
自動再起動タスクの「失敗時に再起動」設定で 1 分後にリトライ × 999 回
停止 / 起動Stop-ScheduledTask / Start-ScheduledTask（プロセスを確実に止めるには Stop-Process も併用）
💡 NSSM 等を使いたい場合 より高機能なサービス管理（services.msc 連携、組込みログローテーション等）が必要な場合は NSSM や WinSW を導入する余地もありますが、本手順では 追加インストール 0 を優先。

2.4ディレクトリ作成

PS> New-Item -ItemType Directory -Force C:\Apps\WbsWeb | Out-Null
PS> New-Item -ItemType Directory -Force C:\AppsData\WbsWeb\logs | Out-Null
PS> New-Item -ItemType Directory -Force C:\AppsData\WbsWeb\templates | Out-Null
PS> New-Item -ItemType Directory -Force C:\AppsData\WbsWeb\addins | Out-Null
📌 templates と addins ディレクトリ
  • templates\ ← Excel 出力／取込で使う旧テンプレ (テンプレートファイル.xls) を置く（4.3 で配置）
  • addins\ ← km_module.xla（ガント描画用 Excel アドイン）を置く。/api/downloads/km_module.xla エンドポイントから利用者に配布される（4.4 で配置）

DEV3. 開発機側でビルド

このマシン（c:\Git\WBS\WbsWeb\）で本番用成果物を作成します。

PS> cd c:\Git\WBS\WbsWeb

# 1) Vue 本番ビルド
PS> pnpm --filter web build
→ packages\web\dist\ が生成される

# 2) Vue 成果物を API の public へ同期
PS> robocopy packages\web\dist packages\api\public /MIR

# 3) NestJS 本番ビルド
PS> pnpm --filter api build
→ packages\api\dist\ が生成される

# 4) 本番依存だけまとめた配布ディレクトリを生成
PS> pnpm --filter api --prod deploy C:\tmp\WbsWeb-deploy
💡 pnpm deploy pnpm の本番デプロイ専用コマンド。node_modules と dist と public を含む完結した配布ディレクトリを C:\tmp\WbsWeb-deploy に作ります。lockfile を尊重するので開発機と同じ依存バージョンになります。

3.5public のコピー漏れを補正

pnpm deploy は src の public/ を拾わないことがあります。確認＆補正：

PS> robocopy packages\api\public C:\tmp\WbsWeb-deploy\public /MIR

共通4. ファイル転送（アプリ / DB / テンプレ / アドイン）

4.1アプリを本番マシンへ転送

下記いずれかの方法で C:\tmp\WbsWeb-deploy の中身を本番マシンの C:\Apps\WbsWeb\ に配置します。

方法例
同一 LAN 上で共有経由 robocopy C:\tmp\WbsWeb-deploy \\<本番IP>\C$\Apps\WbsWeb /MIR /XD logs
※ 本番マシンの管理共有 C$ が有効＆管理者認証が通る前提
USB / ファイルサーバ経由 C:\tmp\WbsWeb-deploy 全体を zip 化して持ち運び、本番側で C:\Apps\WbsWeb に展開

4.2既存 SQLite データを持ち込み

DEV開発機の SQLite ファイルを本番側に配置：

# 開発機側で、SQLite を一度安全にスナップショット（推奨）
PS> cd c:\Git\WBS\WbsWeb\packages\api
PS> node -e "const D=require('better-sqlite3');const d=new D('data\\wbs.db',{readonly:true});d.exec('VACUUM INTO ''C:\\tmp\\wbs-export.db''');d.close()"
# C:\tmp\wbs-export.db ができる

PROD本番側に配置：

PS> # 開発機から C:\tmp\wbs-export.db を本番マシンへ転送した後
PS> Copy-Item C:\tmp\wbs-export.db C:\AppsData\WbsWeb\wbs.db
📌 VACUUM INTO とは SQLite を停止せずに一貫性のあるコピーを取る公式の方法です。copy でファイル直コピーすると WAL ファイルとズレが生じて壊れる場合があるため、必ずこちらを使ってください。

4.3Excel テンプレートを配置

Excel 出力／取込で使うレガシーテンプレートを本番側に配置します。

# 開発機から本番マシンへ転送した後
PS> Copy-Item C:\tmp\テンプレートファイル.xls C:\AppsData\WbsWeb\templates\テンプレートファイル.xls
📌 テンプレートのソース 開発機側では c:\Git\WBS\テンプレートファイル.xls が原本です。 WbsWeb リポジトリ の外 にあるため、本番転送時はリポジトリと別途コピーが必要です。

4.4Excel アドイン (km_module.xla) を配置

WBS Web の /downloads ページから利用者にアドインを配布するため、ファイルを置きます。

# 開発機から本番マシンへ転送した後
PS> Copy-Item C:\tmp\km_module.xla C:\AppsData\WbsWeb\addins\km_module.xla
📌 km_module.xla のソース 開発機側では c:\Git\WBS\AddIns\km_module.xla（約 1.77 MB）が原本です。 本番化後、利用者は http://<サーバ>:5000/downloads からダウンロード → 自分の Excel に登録します。 登録手順はそのページに記載済。

4.5マイグレーションを本番 DB に適用

もしマイグレーションが追加されていた場合に備えて、本番側でも一度実行します（重複適用は安全）：

PS> $env:DB_PATH = "C:\AppsData\WbsWeb\wbs.db"
PS> cd C:\Apps\WbsWeb
PS> node node_modules\drizzle-kit\bin.cjs migrate
PS> Remove-Item Env:DB_PATH

PROD5. 初回サービス登録

5.1MCP トークンを生成（16 文字以上のランダム）

本番運用では MCP_TOKEN 環境変数の設定が必須です。下記で 32 文字のトークンを生成します：

PS> $token = -join ((48..57) + (65..90) + (97..122) | Get-Random -Count 32 | % {[char]$_})
PS> $token
# 例: aB3xK9mP2qR7nW4yL8tZ5jH6cV1fD0sE

# この値をメモしてください（5.2 の start.cmd に埋め込みます）
⚠ MCP_TOKEN の扱い このトークンは MCP エンドポイント（/mcp）の Bearer 認証に使われます。同一マシンからしかアクセスできない仕組み（Origin が localhost / 127.0.0.1 限定）なので外部漏れリスクは小さいですが、平文ログに残さないよう注意してください。

5.2起動スクリプト start.cmd を作成

環境変数の設定とログリダイレクトをまとめたラッパースクリプトを C:\AppsData\WbsWeb\start.cmd として保存します。 5.1 で生成したトークンを MCP_TOKEN= 行に貼り付けてください。 

@echo off
REM WBS Web 起動スクリプト (タスクスケジューラから呼ばれる)
set PORT=5000
set HOST=0.0.0.0
set NODE_ENV=production
set DB_PATH=C:\AppsData\WbsWeb\wbs.db
set XLS_TEMPLATE_PATH=C:\AppsData\WbsWeb\templates\テンプレートファイル.xls
set XLA_ADDIN_PATH=C:\AppsData\WbsWeb\addins\km_module.xla
set MCP_TOKEN=<5.1 で生成した 32 文字を貼る>
cd /d C:\Apps\WbsWeb
"C:\Program Files\nodejs\node.exe" dist\main.js ^
  >> C:\AppsData\WbsWeb\logs\stdout.log ^
  2>> C:\AppsData\WbsWeb\logs\stderr.log
📌 保存方法 メモ帳で作る場合は「ファイル → 名前を付けて保存」で 「ファイルの種類」を「すべてのファイル (*.*)」に切替えて start.cmd として保存（拡張子が .txt にならないように注意）。 文字コードは ANSI / Shift_JIS を選択（UTF-8 だと set 文字列が化ける可能性あり）。

5.3タスクスケジューラに登録

PowerShell（管理者）で実行：

PS> $action = New-ScheduledTaskAction -Execute "C:\AppsData\WbsWeb\start.cmd"
PS> $trigger = New-ScheduledTaskTrigger -AtStartup
PS> $principal = New-ScheduledTaskPrincipal `
        -UserId "SYSTEM" -LogonType ServiceAccount -RunLevel Highest
PS> $settings = New-ScheduledTaskSettingsSet `
        -RestartCount 999 `
        -RestartInterval (New-TimeSpan -Minutes 1) `
        -AllowStartIfOnBatteries `
        -DontStopIfGoingOnBatteries `
        -ExecutionTimeLimit (New-TimeSpan -Seconds 0)
PS> Register-ScheduledTask -TaskName "WbsWeb" `
        -Action $action -Trigger $trigger -Principal $principal -Settings $settings
PS> Start-ScheduledTask -TaskName "WbsWeb"
💡 各パラメータの意味
  • -AtStartup: Windows 起動時に発火（ログオン不要）
  • SYSTEM + Highest: SYSTEM アカウントで最高権限。ファイアウォール開放したポートにバインド可能
  • RestartCount 999 / Interval 1 min: クラッシュ時に 1 分後に再起動 × 最大 999 回（実質無限）
  • ExecutionTimeLimit 0: 実行時間制限なし（永続常駐させる）
  • AllowStartIfOnBatteries / DontStop: ノート PC のバッテリ駆動でも止めない

5.4動作確認（起動直後）

PS> Get-ScheduledTask -TaskName "WbsWeb" | Get-ScheduledTaskInfo

PS> Get-Process node -ErrorAction SilentlyContinue

8. 更新リリース手順（2 回目以降）

機能追加・修正後に本番を更新する手順。マイグレーションが追加されていない場合と、追加されている場合で 1 ステップ違います。

8.1共通: DEV開発機でビルド

PS> cd c:\Git\WBS\WbsWeb
PS> pnpm --filter web build
PS> robocopy packages\web\dist packages\api\public /MIR
PS> pnpm --filter api build
PS> pnpm --filter api --prod deploy C:\tmp\WbsWeb-deploy
PS> robocopy packages\api\public C:\tmp\WbsWeb-deploy\public /MIR

8.2PROD本番マシンでサービス停止 → 差し替え → 起動

# 1) タスクと実プロセスの両方を止める
#    Stop-ScheduledTask は task instance を終了させるが、子プロセス node.exe が
#    残ることがあるので念のため Stop-Process も実行
PS> Stop-ScheduledTask -TaskName "WbsWeb" -ErrorAction SilentlyContinue
PS> Get-CimInstance Win32_Process |
        Where-Object { $_.CommandLine -like "*WbsWeb\dist\main.js*" } |
        ForEach-Object { Stop-Process -Id $_.ProcessId -Force }

# 2) アプリ部分のみ差し替え（C:\AppsData\ は触らない）
#    テンプレ／アドイン (C:\AppsData\WbsWeb\templates / addins) もそのまま温存
PS> robocopy \\<DEV-IP>\C$\tmp\WbsWeb-deploy C:\Apps\WbsWeb /MIR /XD logs

# 3) マイグレーション追加があった場合のみ実行（idempotent なので毎回打っても安全）
PS> $env:DB_PATH = "C:\AppsData\WbsWeb\wbs.db"
PS> cd C:\Apps\WbsWeb
PS> node node_modules\drizzle-kit\bin.cjs migrate
PS> Remove-Item Env:DB_PATH

# 4) タスクを起動
PS> Start-ScheduledTask -TaskName "WbsWeb"
PS> Get-ScheduledTask -TaskName "WbsWeb" | Get-ScheduledTaskInfo
📌 テンプレ／アドインを更新したい場合 テンプレートファイル.xls や km_module.xla を新しい版に差し替えたいときは、 サービスを止めずに C:\AppsData\WbsWeb\templates\ / C:\AppsData\WbsWeb\addins\ のファイルを上書きするだけで OK（読み込み時にディスクから都度参照しているため再起動不要）。
💡 ダウンタイム目安 停止 → 差し替え → マイグレーション → 起動で 1〜2 分。

PROD9. バックアップ

9.1無停止バックアップ（推奨・日次）

PS> $stamp = Get-Date -Format yyyyMMdd_HHmmss
PS> $dst = "C:\Backup\wbs_$stamp.db"
PS> New-Item -ItemType Directory -Force C:\Backup | Out-Null
PS> $env:SRC = "C:\AppsData\WbsWeb\wbs.db"
PS> $env:DST = $dst
PS> cd C:\Apps\WbsWeb
PS> node -e "const D=require('better-sqlite3');const d=new D(process.env.SRC,{readonly:true});d.exec(`VACUUM INTO '${process.env.DST.replace(/\\/g,'\\\\')}'`);d.close();"
PS> Remove-Item Env:SRC, Env:DST

これをタスクスケジューラに毎日 22:00 などで登録するのがおすすめです。

10. トラブルシューティング

症状対処
別 PC からアクセスできない（localhost からは OK） ① ファイアウォール（6.1 のルール）が有効か
② start.cmd に set HOST=0.0.0.0 が含まれているか
③ netstat -ano | findstr :5000 で 0.0.0.0:5000 リッスンしているか
node.exe が起動しない / すぐ落ちる C:\AppsData\WbsWeb\logs\stderr.log を確認。最頻パターン:
・MCP_TOKEN environment variable must be set → 5.2 の start.cmd に set MCP_TOKEN=... が無い、または 16 文字未満
・SQLITE_CANTOPEN → DB_PATH のパスが間違い／フォルダ権限不足
「壊れた DB」エラー WAL モードでサービス停止せずファイル直コピーした疑い。9.2 の復旧手順で直近バックアップから戻す
更新後、画面が古いまま ブラウザのハードリロード（Ctrl + Shift + R）。それでも変わらないなら C:\Apps\WbsWeb\public\ に新しい index.html がコピーされているか確認
タスクスケジューラで「タスクは終了しました (0x41306)」 子プロセス (node.exe) が異常終了したサイン。stderr.log を確認。
多くは MCP_TOKEN 未設定 or start.cmd 内のパス誤り。
タスクスケジューラで「実行中」だが node.exe が見えない ラッパー (start.cmd) は終了しているが子の node.exe が起動していない。
start.cmd 内のパス（C:\Program Files\nodejs\node.exe 等）が正しいか確認。
npm.ps1 / pnpm.ps1 を読み込むことができません PowerShell の ExecutionPolicy が Restricted のため Node 同梱の .ps1 が実行できない。下記を 1 回叩いて PowerShell を再起動:
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
または npm.cmd / pnpm.cmd を直接叩く（一時回避）
タスクを削除して再登録したい Unregister-ScheduledTask -TaskName "WbsWeb" -Confirm:$false で登録解除 → 5.3 をやり直し
Excel 出力で 500 エラー XLS_TEMPLATE_PATH が指すファイルが存在しているか確認:
Test-Path "C:\AppsData\WbsWeb\templates\テンプレートファイル.xls"
false なら 4.3 でテンプレを再配置。
環境変数も start.cmd を開いて set XLS_TEMPLATE_PATH=... 行を確認
/downloads から km_module.xla がダウンロードできない (404) XLA_ADDIN_PATH が指すファイルが存在しているか確認:
Test-Path "C:\AppsData\WbsWeb\addins\km_module.xla"
false なら 4.4 でアドインを再配置
Excel で .xls を開いてもガントが描画されない 利用者側でアドイン未登録の可能性大。
① /downloads から km_module.xla をダウンロード
② Excel → ファイル → オプション → アドイン → 設定 → 参照 で km_module.xla を追加
③ Excel を再起動して、ファイル開き直し
🚫 やってはいけないこと
  • サービス起動中に C:\AppsData\WbsWeb\wbs.db を直接コピー／編集（壊れる）
  • 更新リリース時の robocopy /MIR でうっかり C:\AppsData\ を含める（永続データ・テンプレ・アドインが消える）
  • C:\Apps\WbsWeb\ 配下にテンプレ／アドインを置く（更新時に消える）。必ず C:\AppsData\WbsWeb\ 側に置く
  • HOST 未設定で 0.0.0.0 にバインドされない状態で「動かない」と騒ぐ

11. セキュリティ方針（現状と将来）

11.1現状（v0.1）

本バージョンは 同一 LAN 内のクローズドネットワーク利用を前提に、認証なしで動作します。

エンドポイント認証到達範囲
/api/*（Web API: プロジェクト・タスク・社員・顧客・休日）なしサーバに到達できる全ホスト
/mcp（MCP: AI ツール連携用）なし同上
静的ファイル（Vue SPA）なし同上
⚠ 同一 LAN 上であれば誰でも操作可能 この前提が崩れる場面（VPN 越し他拠点接続、無線 LAN ゲスト相乗り、共有 PC 利用、リモートワーク経由のトンネリング等）では 第三者がデータの読み書きを行えてしまいます。本番マシンに到達できるネットワーク範囲は、運用責任者が把握・管理してください。

11.2現在の防御線

  • HOST=0.0.0.0 によりサーバ自身は LAN に開いているが、ファイアウォール（6.1）で TCP 5000 を社内 LAN セグメントのみに絞ることで、不要なネットワークからの到達を遮断する
  • HTTP（平文）。社内ネットワーク内の盗聴・改ざんは想定外
  • 同時編集なし。複数人が同じプロジェクトを同時に書き換えると最後勝ち＝データ競合のリスクあり（仕様）

11.3将来の認証導入計画（TODO）

下記の要件が発生した時点で、認証層を導入します。Web API と MCP は同じ仕組みで守る方針です（個別に部分認証はしない）。

トリガー必要対応
VPN や別拠点からの利用が始まる 少なくとも Basic 認証 / IP 制限 / リバースプロキシ前段で守る
社外公開 / クラウド配置 HTTPS 化 + 本格的なログイン認証（NestJS Passport で JWT or セッション）
MCP 側は McpAuthGuard（コード上に保持済、現在コメントアウト）を再有効化し、MCP_TOKEN 環境変数を必須に戻す
ユーザ別の操作ログ（誰がいつ何を変えたか）が必要 認証導入と同時に、変更履歴テーブルへ user_id 列を追加して記録
