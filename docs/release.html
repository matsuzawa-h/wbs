PROD2. 本番マシン側 ワンタイム準備

初回のみ、本番マシン上で以下を実施します。すべて管理者として PowerShell を開いて実行してください。

2.1Node.js 22 LTS のインストール

⚠ 開発機と本番マシンで Node.js のメジャーバージョンを必ず一致させる better-sqlite3 はネイティブモジュール (C++) で、Node のメジャーバージョンごとに 別の .node バイナリが必要です。バージョンがズレると起動時に NODE_MODULE_VERSION 127 vs 137 のような不一致エラーが出て起動できません。
本書は開発機が Node.js 22 LTS を使用している前提で書かれています。 本番側も 22 LTS に揃えてください。
  1. ブラウザで https://nodejs.org/dist/latest-v22.x/ を開く
    （または https://nodejs.org/dist/v22.20.0/node-v22.20.0-x64.msi を直接 DL）
  2. 22 系の最新 Windows Installer（.msi）をダウンロード
  3. インストーラーを起動 → 既定設定で完了（PATH 追加にチェックが入っていることを確認）
  4. PowerShell を一度閉じて開き直し、確認: 
    PS> node --version
v22.x.x   ← 必ず "v22." で始まる
PS> npm --version
10.x.x
📌 Node 24 など別バージョンを入れてしまった場合 「設定 → アプリ → インストールされているアプリ → Node.js」からアンインストール後、22 LTS の MSI をインストールし直してください。

2.2pnpm を有効化

PS> corepack enable
PS> pnpm --version
9.x.x
💡 corepack Node.js 16.10 以降に同梱されている公式のパッケージマネージャ管理ツールです。pnpm を別途インストールせずに使えます。
⚠ npm.ps1 を読み込むことができません エラーが出る場合 Windows のクリーン環境では PowerShell の ExecutionPolicy が Restricted になっていて、Node.js 同梱の npm.ps1 / pnpm.ps1 が実行できません。下記を 1 回実行 → PowerShell を再起動してください: 
PS> Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
確認プロンプトには Y で答えます。-Scope CurrentUser なので、サービスを実行する SYSTEM アカウントには影響しません（タスクスケジューラから直接 node.exe を呼ぶため ExecutionPolicy 非依存）。

2.3常駐方式：Windows タスクスケジューラ

追加インストールなしで Node.js プロセスを常駐させるため、Windows 標準のタスクスケジューラを使います。具体的な設定は 5 章で実施。

項目方式
起動トリガシステム起動時（At startup、ユーザログオン不要）
実行アカウントSYSTEM
環境変数 / ログリダイレクトラッパー start.cmd 経由（5.1 で作成）
自動再起動失敗時に 1 分後にリトライ × 999 回
停止 / 起動Stop-ScheduledTask + Stop-Process / Start-ScheduledTask

PS> New-Item -ItemType Directory -Force C:\AppsData\WbsWeb\templates | Out-Null

📌 templates と addins ディレクトリ

  • addins\ ← km_module.xla（ガント描画用 Excel アドイン）を置く。/api/downloads/km_module.xla エンドポイントから利用者に配布される（4.4 で配置）

DEV3. 開発機側でビルド

このマシン（c:\Git\WBS\WbsWeb\）で本番用成果物を作成します。

3.1初回のみ: pnpm install

リポジトリの .npmrc で node-linker=hoisted を指定しているため、 初回または .npmrc 更新後は 1 回 pnpm install を実行して node_modules をフラット構造に作り直します。これによりシンボリックリンクが使われず、 本番マシンへ robocopy で転送しても依存解決が壊れません。 

PS> cd c:\Git\WBS\WbsWeb
PS> pnpm install

3.2Web / API ビルドと配布ディレクトリ生成

PS> cd c:\Git\WBS\WbsWeb

# 1) Vue 本番ビルド → packages\web\dist\
PS> pnpm --filter web build

# 2) Vue 成果物を API の public へ同期
PS> robocopy packages\web\dist packages\api\public /MIR

# 3) NestJS 本番ビルド → packages\api\dist\
PS> pnpm --filter api build

# 4) 配布ディレクトリ（前回ぶんがあれば削除してから生成）
PS> Remove-Item C:\tmp\WbsWeb-deploy -Recurse -Force -ErrorAction SilentlyContinue
PS> pnpm --filter api --prod deploy C:\tmp\WbsWeb-deploy

# 5) public のコピー漏れ補正（pnpm deploy は src/public を拾わないことがある）
PS> robocopy packages\api\public C:\tmp\WbsWeb-deploy\public /MIR
💡 ポイント
  • 配布先が空でないと ERR_PNPM_DEPLOY_DIR_NOT_EMPTY エラー → 事前削除 or --force
  • 完了時点で C:\tmp\WbsWeb-deploy\node_modules\tslib が トップレベルに実体として存在するはず（hoisted が効いている証拠）

共通4. ファイル転送（アプリ / DB / テンプレ / アドイン）

4.1アプリを本番マシンへ転送

下記いずれかの方法で C:\tmp\WbsWeb-deploy の中身を本番マシンの C:\Apps\WbsWeb\ に配置します。

方法例
同一 LAN 上で共有経由 robocopy C:\tmp\WbsWeb-deploy \\<本番IP>\C$\Apps\WbsWeb /MIR /XD logs
※ 本番マシンの管理共有 C$ が有効＆管理者認証が通る前提
USB / ファイルサーバ経由 C:\tmp\WbsWeb-deploy 全体を zip 化して持ち運び、本番側で C:\Apps\WbsWeb に展開

4.2既存 SQLite データを持ち込み

DEV開発機の SQLite ファイルを本番側に配置：

# 開発機側で、SQLite を一度安全にスナップショット（推奨）
PS> cd c:\Git\WBS\WbsWeb\packages\api
PS> node -e "const D=require('better-sqlite3');const d=new D('data\\wbs.db',{readonly:true});d.exec('VACUUM INTO ''C:\\tmp\\wbs-export.db''');d.close()"
# C:\tmp\wbs-export.db ができる

PROD本番側に配置：

PS> # 開発機から C:\tmp\wbs-export.db を本番マシンへ転送した後
PS> Copy-Item C:\tmp\wbs-export.db C:\AppsData\WbsWeb\wbs.db
📌 VACUUM INTO とは SQLite を停止せずに一貫性のあるコピーを取る公式の方法です。copy でファイル直コピーすると WAL ファイルとズレが生じて壊れる場合があるため、必ずこちらを使ってください。

4.3Excel テンプレートを配置

Excel 出力／取込で使うレガシーテンプレートを本番側に配置します。

# 開発機から本番マシンへ転送した後
PS> Copy-Item C:\tmp\テンプレートファイル.xls C:\AppsData\WbsWeb\templates\テンプレートファイル.xls
📌 テンプレートのソース 開発機側では c:\Git\WBS\テンプレートファイル.xls が原本です。 WbsWeb リポジトリ の外 にあるため、本番転送時はリポジトリと別途コピーが必要です。

4.4Excel アドイン (km_module.xla) を配置

WBS Web の /downloads ページから利用者にアドインを配布するため、ファイルを置きます。

# 開発機から本番マシンへ転送した後
PS> Copy-Item C:\tmp\km_module.xla C:\AppsData\WbsWeb\addins\km_module.xla
📌 km_module.xla のソース 開発機側では c:\Git\WBS\AddIns\km_module.xla（約 1.77 MB）が原本です。 本番化後、利用者は http://<サーバ>:5000/downloads からダウンロード → 自分の Excel に登録します。 登録手順はそのページに記載済。

4.5マイグレーションを本番 DB に適用

もしマイグレーションが追加されていた場合に備えて、本番側でも一度実行します（重複適用は安全）：

PS> $env:DB_PATH = "C:\AppsData\WbsWeb\wbs.db"
PS> cd C:\Apps\WbsWeb
PS> node node_modules\drizzle-kit\bin.cjs migrate
PS> Remove-Item Env:DB_PATH

4.6操作手順書 (docs) を配置

アプリ内の「操作手順」ページ（/manual）は、サーバが docs\manual.html と docs\screenshots\* を実行時に配信します。 これらは pnpm deploy の成果物（C:\tmp\WbsWeb-deploy）には含まれません （テンプレ／アドインと同様、リポジトリ同梱物を別途配置する必要があります）。 未配置だと「操作手順」ページと /api/help/* が 404（docs file not found）になります。

アプリ差し替え（robocopy /MIR）で消えない永続領域 C:\AppsData\WbsWeb\docs に置き、 起動スクリプトの環境変数 MANUAL_DOCS_DIR でそこを指す方式にします （DB／テンプレ／アドインと同じ思想。start.cmd 側は 5.1 に記載済み）。 

# 開発機から本番マシンへ docs を転送（共有経由の例）
PS> robocopy c:\Git\WBS\WbsWeb\docs \\<本番IP>\C$\AppsData\WbsWeb\docs /MIR
📌 ソースと反映 原本はリポジトリの c:\Git\WBS\WbsWeb\docs\（manual.html ＋ screenshots\）。 手順書を更新したら、この docs を本番の C:\AppsData\WbsWeb\docs へ 再 robocopy するだけで反映されます（サービス再起動不要・ディスク都度参照）。 MANUAL_DOCS_DIR は HelpController が最優先で参照します。

PROD5. 初回サービス登録

5.1起動スクリプト start.cmd を作成

環境変数の設定とログリダイレクトをまとめたラッパースクリプトを C:\AppsData\WbsWeb\start.cmd として保存します。

@echo off
REM WBS Web 起動スクリプト (タスクスケジューラから呼ばれる)
set PORT=5000
set HOST=0.0.0.0
set NODE_ENV=production
set DB_PATH=C:\AppsData\WbsWeb\wbs.db
set XLS_TEMPLATE_PATH=C:\AppsData\WbsWeb\templates\テンプレートファイル.xls
set XLA_ADDIN_PATH=C:\AppsData\WbsWeb\addins\km_module.xla
set MANUAL_DOCS_DIR=C:\AppsData\WbsWeb\docs
REM MCP 認証を再有効化する場合のみ MCP_TOKEN を設定 (11 章参照)。
REM set MCP_TOKEN=<16 文字以上のランダム値>
cd /d C:\Apps\WbsWeb
"C:\Program Files\nodejs\node.exe" dist\main.js ^
  >> C:\AppsData\WbsWeb\logs\stdout.log ^
  2>> C:\AppsData\WbsWeb\logs\stderr.log
📌 MCP_TOKEN について v0.1 では MCP 認証ガード (McpAuthGuard) がコメントアウトされており、 MCP_TOKEN 環境変数は 設定不要です。 将来認証を再有効化する場合の手順は 11 章 セキュリティ方針 に記載。
📌 保存方法（文字コードと改行コードに注意）
  • メモ帳で作る場合は「ファイル → 名前を付けて保存」で 「ファイルの種類」を「すべてのファイル (*.*)」に切替え、 start.cmd として保存（拡張子が .txt にならないように注意）
  • 文字コード：ANSI / Shift_JIS（UTF-8 だと set 文字列が化ける可能性）
  • 改行コード：CRLF 必須。Windows 11 のメモ帳は LF のみで保存することがあり、 その場合 cmd.exe が '000' など意味不明な文字列で 「内部コマンドまたは外部コマンドとして認識されていません」エラーを出す。
確認方法： 
PS> Format-Hex -Path C:\AppsData\WbsWeb\start.cmd | Select-Object -First 3
# 各行末が "0D 0A" になっていれば OK。"0A" だけ なら下記で変換:
PS> (Get-Content C:\AppsData\WbsWeb\start.cmd) |
        Set-Content C:\AppsData\WbsWeb\start.cmd -Encoding OEM -Force
💡 cmd.exe から書く方法（CRLF と ANSI が確実） メモ帳で文字コード／改行コードの罠を避けたい場合、cmd.exe を起動して直接生成： 
cd C:\AppsData\WbsWeb
del start.cmd

(
echo @echo off
echo set PORT=5000
echo set HOST=0.0.0.0
echo set NODE_ENV=production
echo set DB_PATH=C:\AppsData\WbsWeb\wbs.db
echo set XLS_TEMPLATE_PATH=C:\AppsData\WbsWeb\templates\テンプレートファイル.xls
echo set XLA_ADDIN_PATH=C:\AppsData\WbsWeb\addins\km_module.xla
echo set MANUAL_DOCS_DIR=C:\AppsData\WbsWeb\docs
echo cd /d C:\Apps\WbsWeb
echo "C:\Program Files\nodejs\node.exe" dist\main.js ^^^>^^^> C:\AppsData\WbsWeb\logs\stdout.log 2^^^>^^^> C:\AppsData\WbsWeb\logs\stderr.log
) > start.cmd
cmd.exe からの echo > は確実に CRLF + ANSI で書き込まれます。

5.2タスクスケジューラに登録

PowerShell（管理者）で実行：

PS> $action = New-ScheduledTaskAction -Execute "C:\AppsData\WbsWeb\start.cmd"
PS> $trigger = New-ScheduledTaskTrigger -AtStartup
PS> $principal = New-ScheduledTaskPrincipal `
        -UserId "SYSTEM" -LogonType ServiceAccount -RunLevel Highest
PS> $settings = New-ScheduledTaskSettingsSet `
        -RestartCount 999 `
        -RestartInterval (New-TimeSpan -Minutes 1) `
        -AllowStartIfOnBatteries `
        -DontStopIfGoingOnBatteries `
        -ExecutionTimeLimit (New-TimeSpan -Seconds 0)
PS> Register-ScheduledTask -TaskName "WbsWeb" `
        -Action $action -Trigger $trigger -Principal $principal -Settings $settings
PS> Start-ScheduledTask -TaskName "WbsWeb"
💡 各パラメータの意味
  • -AtStartup: Windows 起動時に発火（ログオン不要）
  • SYSTEM + Highest: SYSTEM アカウントで最高権限。ファイアウォール開放したポートにバインド可能
  • RestartCount 999 / Interval 1 min: クラッシュ時に 1 分後に再起動 × 最大 999 回（実質無限）
  • ExecutionTimeLimit 0: 実行時間制限なし（永続常駐させる）
  • AllowStartIfOnBatteries / DontStop: ノート PC のバッテリ駆動でも止めない

5.3動作確認（起動直後）

PS> Get-ScheduledTask -TaskName "WbsWeb" | Get-ScheduledTaskInfo
# LastTaskResult が 267009 (= currently running) であれば OK
PS> Get-Process node -ErrorAction SilentlyContinue
# node.exe プロセスが見える
PS> Get-Content C:\AppsData\WbsWeb\logs\stdout.log -Tail 30

5.4環境変数の意味

変数必須意味本番値
PORT○HTTP リッスンポート5000
HOST○バインドアドレス（必ず 0.0.0.0、これがないと別 PC からつながらない）0.0.0.0
NODE_ENV○NestJS の本番モード切替production
DB_PATH○SQLite ファイルの絶対パスC:\AppsData\WbsWeb\wbs.db
XLS_TEMPLATE_PATH○Excel 出力／取込で使う旧テンプレ テンプレートファイル.xls の絶対パスC:\AppsData\WbsWeb\templates\テンプレートファイル.xls
XLA_ADDIN_PATH○/api/downloads/km_module.xla エンドポイントで配信する Excel アドインの絶対パスC:\AppsData\WbsWeb\addins\km_module.xla
MCP_TOKEN—MCP Bearer 認証用シークレット（16 文字以上）。v0.1 では MCP 認証が無効化されているため設定不要。11 章で再有効化する場合のみ設定未設定
📌 パスの考え方 XLS_TEMPLATE_PATH / XLA_ADDIN_PATH を明示することで、アプリ更新時に C:\Apps\WbsWeb\ 配下を robocopy /MIR で全削除しても、 永続データ側 (C:\AppsData\WbsWeb\) に置いたテンプレ／アドインは無傷で残ります。

PROD6. ファイアウォール開放と動作確認

6.1受信ポート 5000 を許可

PS> New-NetFirewallRule -DisplayName "WbsWeb 5000" `
        -Direction Inbound -Protocol TCP -LocalPort 5000 -Action Allow

6.2本番マシンの IP アドレスを確認

PS> Get-NetIPAddress -AddressFamily IPv4 |
        Where-Object { $_.InterfaceAlias -notmatch "Loopback|vEthernet" } |
        Select-Object IPAddress, InterfaceAlias

6.3動作確認

  1. 本番マシン自身で http://localhost:5000/ を開く → プロジェクト一覧が表示される
  2. 同一 LAN の別 PC のブラウザで http://<本番マシンの IP>:5000/ を開く → 同じ画面
  3. 持ち込んだ既存データ（プロジェクト・顧客・社員）が表示されていることを確認

「壊れた DB」エラー WAL モードでサービス停止せずファイル直コピーした疑い。9.2 の復旧手順で直近バックアップから戻す

11. セキュリティ方針（現状と将来）

11.1現状（v0.1）

本バージョンは 同一 LAN 内のクローズドネットワーク利用を前提に、認証なしで動作します。

エンドポイント認証到達範囲
/api/*（Web API: プロジェクト・タスク・社員・顧客・休日）なしサーバに到達できる全ホスト
/mcp（MCP: AI ツール連携用）なし同上
静的ファイル（Vue SPA）なし同上
⚠ 同一 LAN 上であれば誰でも操作可能 この前提が崩れる場面（VPN 越し他拠点接続、無線 LAN ゲスト相乗り、共有 PC 利用、リモートワーク経由のトンネリング等）では 第三者がデータの読み書きを行えてしまいます。本番マシンに到達できるネットワーク範囲は、運用責任者が把握・管理してください。

11.2現在の防御線

  • HOST=0.0.0.0 によりサーバ自身は LAN に開いているが、ファイアウォール（6.1）で TCP 5000 を社内 LAN セグメントのみに絞ることで、不要なネットワークからの到達を遮断する
  • HTTP（平文）。社内ネットワーク内の盗聴・改ざんは想定外
  • 同時編集なし。複数人が同じプロジェクトを同時に書き換えると最後勝ち＝データ競合のリスクあり（仕様）

11.3将来の認証導入計画（TODO）

下記の要件が発生した時点で、認証層を導入します。Web API と MCP は同じ仕組みで守る方針です（個別に部分認証はしない）。

トリガー必要対応
VPN や別拠点からの利用が始まる 少なくとも Basic 認証 / IP 制限 / リバースプロキシ前段で守る
社外公開 / クラウド配置 HTTPS 化 + 本格的なログイン認証（NestJS Passport で JWT or セッション）
MCP 側は McpAuthGuard（コード上に保持済、現在コメントアウト）を再有効化し、MCP_TOKEN 環境変数を必須に戻す
ユーザ別の操作ログ（誰がいつ何を変えたか）が必要 認証導入と同時に、変更履歴テーブルへ user_id 列を追加して記録

12. MCP 接続手順（開発者・運用者向け）

Claude Code などの MCP 対応クライアントから、本番 WBS Web の DB を 直接読み書きするための接続手順です。利用者画面とは別経路で、 CRUD 系の操作を AI に任せたい / 一括登録を自動化したいといった場面で使います。

12.1エンドポイント情報

項目値
URLhttp://<本番マシン IP>:5000/mcp または http://TG120286:5000/mcp
TransportStreamable HTTP（POST / GET / DELETE）
認証なし（v0.1 では認証ガード無効化中、11 章 参照）
ツール数約 30（Customers / Projects / Tasks / Employees / Holidays の各 CRUD）

12.2クライアント側 .mcp.json の例

このリポジトリには既に c:\Git\WBS\WbsWeb\.mcp.json として下記が配置済みです：

{
  "mcpServers": {
    "wbs-local": {
      "type": "http",
      "url": "http://127.0.0.1:5000/mcp"
    },
    "wbs-prod": {
      "type": "http",
      "url": "http://TG120286:5000/mcp"
    }
  }
}
💡 ホスト名で繋がらない場合は IP 直打ちに切替 TG120286 で名前解決できない環境では、http://192.168.x.x:5000/mcp のように IP を直接書いてください。本番マシンの IP は 6.2 章のコマンドで確認できます。

12.3接続前の疎通確認

PS> Invoke-WebRequest -Uri "http://TG120286:5000/api/health" -UseBasicParsing

PS> Invoke-WebRequest -Uri "http://TG120286:5000/mcp" -UseBasicParsing -Method GET
# 405 Method Not Allowed が返れば MCP endpoint まで到達できている
# (MCP は POST 中心、GET 単独は受け付けないので 405 が正解)
