5. 主要処理設計

5.1営業日による終了日算出

computeEndDate(start, duration) ＝ 開始日から「営業日で duration 日」進めた日。 土日 ＋ holidays を非稼働日として除外。HolidaysService は休日集合を短期キャッシュ（連続再計算の負荷低減）。 レベル3タスク／個人タスクの end_date を保存時に再計算。

5.2連動カスケード（DateCascadeService）

レベル3タスクの開始/期間（＝終了日）が変わると、同一中項目内の後続タスクを 変更した営業日数と同じだけ平行シフトする（直列に詰め直さない）。 連続/重なりで「連動対象」を判定し、間に余裕がある独立タスクはシフト範囲で停止。 画面の「連動」OFF（cascade=false）時は後続を動かさず祖先再集計のみ。 

例) 5/15・5/15・5/15（各1日, 同中項目）で先頭を 5/16 へ
  旧(直列)  : 5/16 / 5/17 / 5/18
  新(平行)  : 5/16 / 5/16 / 5/16   ← +1営業日を全後続に適用

5.3集計ロールアップ

recomputeAllAncestors：中(2)→大(1) の順で、子タスクから 予定開始＝min、予定終了＝max、予定日数＝営業日換算、実績開始＝最小、 実績終了＝全子完了時のみ確定、工数＝合算 を自動再計算。タスクの追加/更新/削除/複製/並び替え後に実行。

5.4状態の自動判定（導出・DB 非保存）

状態は computeStatus(task, today) で算出（保存しない）。 actual_end_date があれば「完了」（予定通り/前倒し/遅れの内訳付）、 actual_start_date のみなら「実行中」または期限超過で「遅延中」、 未着手は開始予定との関係で「未着手／着手遅れ」。 画面の「✓ 完了にする」は実績終了日＝今日（＋進捗100%）を書き込み、結果として状態が「完了」になる。

5.5タスク複製

対象タスクとその配下を preorder でディープコピー（id 再採番・parent_id 再マッピング）。 進捗・実績も引継ぎ、レベル3は終了日再算出。sort_order を採番し直して 元の部分木の直後に新ブロックを配置、祖先を再集計。

5.6並び替え（reorder）

PUT /api/tasks/reorder で {id, sortOrder} 群を一括更新。表示は (project_id, sort_order) 昇順（idx_wbs_tasks_sort）。階層整合はアプリ側で担保。

5.7個人タスクの分離

personal_tasks は WBS ツリーに属さない。プロジェクト紐付けは任意（レポート文脈用、 project 削除で SET NULL）。ガント／Excel 出力には一切出さず、担当別予定（一覧・カレンダー）でのみ 社員の WBS 割当と並べて表示・編集する。終了日は 5.1 で算出。

5.8Excel 取込（プレビュー → 確定）

preview はアップロード xls を解析して階層・日付・工数等を抽出（DB 変更なし）。 commit で対象プロジェクトの wbs_tasks を解析結果に置換 （既存削除＋新規作成）、その後 5.1〜5.3 を適用。出力は旧テンプレートへ BIFF 直書き。

5.9表示メンバー（ソフト除外）

project_members は各プロジェクトの担当ピッカー候補。メンバーから外しても 既存タスクの assignee_id は保持（ソフト除外）し、UI は「(メンバー外)」表示で 新規割当を抑止。PUT /:id/members はメンバー集合を置換。

5.10稼働管理表 CSV 取込（プレビュー → 確定）

別システム出力の Shift-JIS CSV を Node 組込 TextDecoder('shift_jis') で デコード（UTF-8 BOM 付きなら UTF-8 として読む＝再保存フォールバック）。 会計年度（4月始まり）で月→YYYY-MM 変換（1〜3月は翌暦年）。 行区分: 合計 の派生行は破棄、件名「月基準時間」のみ manhour_capacities へ、 zz は非稼働 entry（project なし）、AFT/MNT/SY/空は案件工数。 同一 担当者×CD×作業区分×年月 は合算。 preview は担当者名・顧客名(E列)・プロジェクトCD を名寄せ（DB 変更なし）。 commit は 1 トランザクションで、未一致担当者を社員マスタに、未一致顧客を顧客マスタに自動作成 （顧客は NFKC＋空白除去＋会社表記ゆれ＋小文字化の正規化キーで突合し、表記揺れ・再取込でも 重複作成しない＝既存を再利用。安全側で見た目違いのみ同一視）。 プロジェクト化対象は「作業区分=AFT または SY」かつ「プロジェクトCD有り」の行のみ （CD無しのAFT/SY・MNT/空/zz は工数のみ＝件名ラベル計上、projects 化しない。 ただし同一CDが対象行で既にプロジェクト化されていれば対象外区分の工数もそのプロジェクトへ合算）。 案件の突合は project_codes（CD一意）が正本。登録済CDは常に所属プロジェクトへ（再取込・付け替え防止）。 未登録CDは件名から期間/工程表記を除いた束ね名(stem)で自動グルーピング提案し、 同一 stem の複数CD（工程別など）を 1 プロジェクト＝1ガントに束ねて作成（newGroup）。 既存プロジェクトへ link した場合もそのCDを project_codes に恒久登録し次回以降自動で同じプロジェクトへ。 顧客は名寄せ結果で紐づけ。CD無し（担当者のフリー作業）は既定で「ラベルのみ」＝ projects マスタを作らず project_id=NULL＋label=件名 で計上（zz も同様にラベル保持）。 毎回新規バッチとして明細・月基準時間を投入（履歴保持）。 集計は「選択/最新の取込バッチ ∪ 手入力(batch_id NULL)」で行い、 稼働率は Σhours / base_hours を都度算出（CSV 稼働率行は不使用）。

5.11組織の自動解決（取込で作成エンティティに自動紐付け）

CSV の A列「組織コード」と B列「組織名称」を preview で抽出し、code 一致で既存 organizations を探す。プレビュー応答 organizationMatch で「既存に紐付け／新規作成／紐付けない（skip）」を選択でき、 commit は最初に resolvedOrgId を確定し、その取込で新規作成される担当者・顧客・仮案件の organization_id に設定する（既存エンティティの organization_id は 変更しない＝touched しない）。 新規作成時は code 重複（再取込）を防ぐため、code 一致の既存組織があれば再利用。 組織は任意深さの自己参照階層（parent_id）で、循環はアプリ層で防止する。 過去取込（0011 以前）の既存エンティティは organization_id=NULL のまま残るため、必要であれば 組織マスタ画面（/organizations）と各マスタ画面のフィルタ／編集ダイアログから手動で割当する。
