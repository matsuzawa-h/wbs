customers ──< projects ──< wbs_tasks >── assignees
                 │   │            (assignee_id, ON DELETE SET NULL)

                 │
                 ├─< project_members >── assignees   (PK: project_id+employee_id)
                 └─< personal_tasks   >── assignees   (employee 必須 / project 任意)

                        └─< manhour_capacities >── assignees

2. テーブル定義

型は SQLite の格納クラス。日付列は TEXT（YYYY-MM-DD）、created_at は INTEGER（unixepoch 秒）。PK=主キー / FK=外部キー。

2.1 customers（顧客マスタ）
列型 / 既定制約・説明
idINTEGERPK / AUTOINCREMENT
codeTEXT顧客コード（任意）
nameTEXTNOT NULL 顧客名
contact_name / contact_email / contact_phone / addressTEXT連絡先（任意）
is_activeINTEGER / 1NOT NULL 有効フラグ
noteTEXT備考
sort_orderINTEGER / 0NOT NULL 表示順（idx_customers_sort）
created_atINTEGER / unixepoch()NOT NULL 作成時刻
2.2 projects（プロジェクト）
列型 / 既定制約・説明
idINTEGERPK / AUTOINCREMENT
customer_idINTEGERFK customers.id（任意）/ ON DELETE SET NULL（idx_projects_customer）
nameTEXTNOT NULL プロジェクト名
project_codeTEXT別システムのプロジェクトCD（任意・非ユニーク。取込工数の突合キー。idx_projects_code）
is_provisionalINTEGER / 0NOT NULL 仮プロジェクトフラグ（1=仮。稼働見通しで確定/仮を区別）
created_atINTEGER / unixepoch()NOT NULL

各機能（REST エンドポイント群）がどのテーブルを Create / Read / Update / Delete するか。MCP ツールは同一サービス層を呼ぶため CRUD は対応する機能行と同じです。 括弧は FK や連動による副作用。

  • 担当別予定※：割当取得 API は wbs_tasks が対象。個人タスクは別 API（個人タスク行）で取得し画面で統合。

  • MCP ツール（customer_*/project_*/task_*/personal_task_*/employee_*/holiday_*）は対応機能行と同一の CRUD。

5. 主要処理設計

5.1営業日による終了日算出

computeEndDate(start, duration) ＝ 開始日から「営業日で duration 日」進めた日。 土日 ＋ holidays を非稼働日として除外。HolidaysService は休日集合を短期キャッシュ（連続再計算の負荷低減）。 レベル3タスク／個人タスクの end_date を保存時に再計算。

5.2連動カスケード（DateCascadeService）

レベル3タスクの開始/期間（＝終了日）が変わると、同一中項目内の後続タスクを 変更した営業日数と同じだけ平行シフトする（直列に詰め直さない）。 連続/重なりで「連動対象」を判定し、間に余裕がある独立タスクはシフト範囲で停止。 画面の「連動」OFF（cascade=false）時は後続を動かさず祖先再集計のみ。 

例) 5/15・5/15・5/15（各1日, 同中項目）で先頭を 5/16 へ
  旧(直列)  : 5/16 / 5/17 / 5/18
  新(平行)  : 5/16 / 5/16 / 5/16   ← +1営業日を全後続に適用

5.3集計ロールアップ

recomputeAllAncestors：中(2)→大(1) の順で、子タスクから 予定開始＝min、予定終了＝max、予定日数＝営業日換算、実績開始＝最小、 実績終了＝全子完了時のみ確定、工数＝合算 を自動再計算。タスクの追加/更新/削除/複製/並び替え後に実行。

5.4状態の自動判定（導出・DB 非保存）

状態は computeStatus(task, today) で算出（保存しない）。 actual_end_date があれば「完了」（予定通り/前倒し/遅れの内訳付）、 actual_start_date のみなら「実行中」または期限超過で「遅延中」、 未着手は開始予定との関係で「未着手／着手遅れ」。 画面の「✓ 完了にする」は実績終了日＝今日（＋進捗100%）を書き込み、結果として状態が「完了」になる。

5.5タスク複製

対象タスクとその配下を preorder でディープコピー（id 再採番・parent_id 再マッピング）。 進捗・実績も引継ぎ、レベル3は終了日再算出。sort_order を採番し直して 元の部分木の直後に新ブロックを配置、祖先を再集計。

5.6並び替え（reorder）

PUT /api/tasks/reorder で {id, sortOrder} 群を一括更新。表示は (project_id, sort_order) 昇順（idx_wbs_tasks_sort）。階層整合はアプリ側で担保。

5.7個人タスクの分離

personal_tasks は WBS ツリーに属さない。プロジェクト紐付けは任意（レポート文脈用、 project 削除で SET NULL）。ガント／Excel 出力には一切出さず、担当別予定（一覧・カレンダー）でのみ 社員の WBS 割当と並べて表示・編集する。終了日は 5.1 で算出。

5.8Excel 取込（プレビュー → 確定）

preview はアップロード xls を解析して階層・日付・工数等を抽出（DB 変更なし）。 commit で対象プロジェクトの wbs_tasks を解析結果に置換 （既存削除＋新規作成）、その後 5.1〜5.3 を適用。出力は旧テンプレートへ BIFF 直書き。

5.9表示メンバー（ソフト除外）

project_members は各プロジェクトの担当ピッカー候補。メンバーから外しても 既存タスクの assignee_id は保持（ソフト除外）し、UI は「(メンバー外)」表示で 新規割当を抑止。PUT /:id/members はメンバー集合を置換。

5.10稼働管理表 CSV 取込（プレビュー → 確定）

別システム出力の Shift-JIS CSV を Node 組込 TextDecoder('shift_jis') で デコード（UTF-8 BOM 付きなら UTF-8 として読む＝再保存フォールバック）。 会計年度（4月始まり）で月→YYYY-MM 変換（1〜3月は翌暦年）。 行区分: 合計 の派生行は破棄、件名「月基準時間」のみ manhour_capacities へ、 zz は非稼働 entry（project なし）、AFT/MNT/SY/空は案件工数。 同一 担当者×CD×作業区分×年月 は合算。 preview は担当者名・プロジェクトCD を名寄せ（DB 変更なし）。 commit は 1 トランザクションで、未一致担当者を社員マスタに自動作成、 未一致/CD空案件を仮プロジェクト作成（再取込時は CD/名前で既存仮案件を再利用＝重複防止）、 毎回新規バッチとして明細・月基準時間を投入（履歴保持）。 集計は「選択/最新の取込バッチ ∪ 手入力(batch_id NULL)」で行い、 稼働率は Σhours / base_hours を都度算出（CSV 稼働率行は不使用）。
