1. 目的とスコープ

従来 Excel テンプレート（テンプレートファイル.xls）で運用していた WBS／ガント工程表を Web 化し、 複数プロジェクトの工程・進捗・実績・担当を一元管理する。Excel との相互運用（取込・出力）と、 プロジェクトに紐づかない「個人タスク」管理、社員横断の「担当別予定」表示・カレンダーを提供する。 インターネット公開は不要で、社内 LAN 内の 1 台に常駐させる構成とする。

2. 全体構成

フロントエンド（Vue 3 SPA）はビルド成果物を API に同梱し、単一の Node.js プロセス （NestJS）が「静的 SPA 配信」「REST API /api/*」「MCP /mcp」「業務ファイル配信」を兼ねる。 データは同一マシンの SQLite ファイル 1 個。永続データはアプリ本体と別ディレクトリに置き、 アプリ差し替え時に消えないようにする。

5. フロントエンド構成（Vue 3 SPA）

ルート画面概要
/プロジェクト一覧顧客でグルーピング、作成・Excel 取込導線。
/projects/:id/ganttWBS／ガント3 階層（大/中/項目）テーブル＋ガント。連動・複製・備考・実績・状態。
/assignments担当別予定社員横断の割当＋個人タスク。一覧／カレンダー切替（既定カレンダー）。
/manhours稼働見通し担当者×月の工数サマリー（確定/仮トグル・稼働率色分け・案件内訳）。CSV取込／仮案件導線。
/projects/:id/manhours案件別工数1 案件の担当者×月。工程表(ガント)と相互リンク。セル手入力（仮）。
/customers / /employees / /holidays各マスタ顧客／社員／休日の保守。
/downloadsダウンロードExcel アドイン取得手順。
/manual操作手順アプリ枠内に /api/help/manual.html を iframe 表示。

状態管理は Pinia（projects / tasks / employees / customers / holidays / projectMembers / manhours ストア）。 ガントは frappe-gantt を土台に、スクロール同期・固定ヘッダ・今日マーカー・週末/休日表示・ バーラベル可読性などを独自実装。担当別予定カレンダーは月グリッド（バーのレーン配置・D&D・新規作成ポップアップ）を独自実装。

6. リクエスト経路

  • 静的 SPA：/ ほか未マッチ URL → packages/api/public の Vue ビルド成果物（useStaticAssets, fallthrough）。ディープリンクは SPA 側ルータが処理。
  • REST：/api/* を各コントローラが処理。検証は class-validator DTO（whitelist）。
  • MCP：/mcp（Streamable HTTP・statelessMode）。ツールは REST と同一のサービス層を呼ぶため挙動が一致。
  • 業務ファイル：Excel 出力／取込、アドイン配布、操作手順・スクショ配信。docs/ は実行時にディスク参照（更新は再起動不要）。

7. データ永続化

  • SQLite 単一ファイル。接続は better-sqlite3、クエリは drizzle-orm。WAL モード。
  • DB パスは環境変数 DB_PATH（本番 C:\AppsData\WbsWeb\wbs.db）。未指定時は cwd/data/wbs.db。
  • スキーマ変更は drizzle-kit マイグレーション（src/db/migrations + _journal.json）。冪等。詳細は別紙「テーブル定義・処理設計書」。
  • 無停止スナップショットは VACUUM INTO（WAL とズレないコピー）。

  • 単一プロセス常駐：Windows タスクスケジューラ（システム起動時・失敗時リトライ）。start.cmd が環境変数とログを設定。
  • ディレクトリ分離：C:\Apps\WbsWeb（毎回 /MIR で総入替）と C:\AppsData\WbsWeb（DB／テンプレ／アドイン／docs＝不可侵）。
  • pnpm hoisted：node_modules がフラット実体 → robocopy/zip 転送で壊れない（本番ネット制限環境でも install 不要）。
  • 長パス対策：配布 zip の展開は標準 tar（エクスプローラ/Expand-Archive は 260 文字超で失敗）。
  • 手順詳細は別紙「リリース手順書（release.html）」8.0 章。

9. 横断的な処理方針

方針概要
営業日による終了日算出開始日＋日数（営業日）から終了日を算出。土日＋休日マスタを非稼働日として除外。
連動カスケードレベル3タスクの日付変更時、同一中項目内の後続を同じ営業日数だけ平行シフト（直列再配置しない）。余裕のある独立タスクは動かさない。連動 OFF も可。
集計ロールアップ大/中項目の予定・実績・工数は子の min/max/合算で自動再計算。
状態の自動判定状態（完了/実行中/遅延中/着手遅れ/未着手）は開始/終了/実績/今日から導出（DB 非保存）。
タスク複製部分木をディープコピー（進捗・実績も引継ぎ）、元の直後に配置、終了日再計算・祖先再集計。
個人タスク分離個人タスクは WBS ツリー外。ガント／Excel に出さず、担当別予定のみで管理。
Excel 入出力旧テンプレートへ BIFF 直書き出力／Excel から WBS 解析・取込（プレビュー→確定）。
稼働見通し稼働管理表 CSV を取込み、担当者×月の工数を集計。会計年度4月始まり（1〜3月=翌暦年）。再取込は毎回新規バッチ（履歴保持）、未一致案件は仮プロジェクト自動作成（CD/名前で重複防止）。確定(取込)/仮(手入力)を区別表示、稼働率＝Σ工数/月基準時間を都度算出。
Shift-JIS デコードCSV は Node 組込 TextDecoder('shift_jis')（full-ICU 既定。追加依存なし）。UTF-8 BOM 付きは UTF-8 として読む（再保存フォールバック）。小ICUビルドは「UTF-8 で再保存」を促す。
MCP 整合MCP ツールは REST と同一サービス層。Zod ↔ DTO パリティをテストで担保。

10. セキュリティ方針

⚠ 認証なし（社内 LAN 前提） Web API・MCP とも現状無認証で動作（社内 LAN クローズド・HTTP 平文）。 MCP 認証ガード（McpAuthGuard）はコメントアウト済み。将来インターネット接続や認証導入時は Web API と MCP を同じ仕組みで保護し、MCP_TOKEN を必須化する方針（release.html セキュリティ方針章参照）。
  • Help 配信はファイル名を検証しパストラバーサルを遮断（不正は 404）。
  • 入力は class-validator（whitelist）で検証。SQL は drizzle のパラメタライズ。

11. 非機能・制約

  • 規模：〜10 ユーザ／社内 LAN。同時編集は非対応（同一プロジェクトを複数人で同時編集しない運用前提）。
  • 可用性：タスクスケジューラで OS 起動時自動起動・失敗時リトライ。ダウンタイムは更新時 1〜2 分。
  • バックアップ：VACUUM INTO による無停止日次スナップショット（release.html バックアップ章）。
  • 互換：開発機と本番で Node.js メジャー一致必須（better-sqlite3 ネイティブ整合）。
